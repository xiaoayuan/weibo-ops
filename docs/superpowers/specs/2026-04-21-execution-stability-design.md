# 执行链路稳定性设计

## 目标

在不推翻现有 `executor + runner` 架构的前提下，为项目补齐一套统一的执行稳定性框架，覆盖当前已存在和已预留的业务动作链路：`CHECK_IN`、`LIKE`、`COMMENT_LIKE`、`REPOST`、`POST`。

本次目标不是新增更多业务动作，而是把现有执行结果从“基本可跑”提升到“结果可信、失败可定位、规则一致、后续可扩展”。

## 当前问题

从现有代码和最近提交看，项目已经具备执行器、编排任务、任务步骤和日志能力，但执行稳定性仍然主要依赖各链路分散处理，存在以下风险：

- 不同动作的成功判定标准不完全一致
- 重试逻辑主要集中在部分链路，缺少统一规则
- 任务、账号、步骤三级状态回写方式不统一
- 日志已有基础信息，但对排障仍不够直接
- 新动作接入时容易复制已有分散逻辑，继续放大维护成本

这些问题会直接影响业务判断：执行到底是成功、部分成功、还是假成功；失败时应该重试、停止、还是人工介入；日志能否支撑排障和复盘。

## 范围

### 纳入本次改造的动作

- `CHECK_IN`
- `LIKE`
- `COMMENT_LIKE`
- `REPOST`
- `POST`

说明：

- `POST` 本次纳入统一稳定性框架，但只定义接入标准和调用约束，不要求本轮落地完整真实发帖能力
- 已有 `mock executor` 与 `weibo executor` 均需兼容
- 已有计划执行、互动执行、编排任务执行均需复用统一执行语义

### 不纳入本次改造的内容

- 新的业务动作类型
- 分布式队列和外部任务调度系统
- 更复杂的风控随机化策略
- 页面级大改版
- 对数据库模型做颠覆式调整

## 设计原则

- 保持现有 `executor + runner` 架构不变，只做最小必要抽象
- 所有动作都走统一的“解析 -> 预检 -> 请求 -> 判定 -> 回写 -> 记录”流程
- 动作差异通过动作配置和动作级处理函数表达，不在多个 runner 中重复散落
- 成功判定必须可解释，日志中要能看到判定依据
- 只对可重试错误做有限重试，避免无意义重复请求

## 统一执行语义

本次为所有动作统一以下 6 个能力。

### 1. 目标解析

为每个动作提供统一的目标解析入口，将外部输入整理为动作执行真正依赖的结构化目标。

示例：

- `CHECK_IN`：解析超话 URL、话题对象 ID、签到端点候选
- `LIKE`：解析微博详情链接中的状态 ID
- `COMMENT_LIKE`：解析评论链接中的评论 ID、微博 ID 或请求目标参数
- `REPOST`：解析目标微博状态 ID、必要时补充原始 UID
- `POST`：解析发帖内容、超话上下文、回调参数要求

输出统一为结构化对象，至少包括：

- `actionType`
- `normalizedTarget`
- `resolvedIdentifiers`
- `warnings`

解析失败时直接返回结构化错误，不进入请求执行。

### 2. 动作级预检

在现有账号登录态预检基础上，为不同动作增加统一动作预检入口，避免每条链路自行判断。

预检最少包含：

- 账号是否可用
- 登录态是否允许当前动作
- 目标解析是否成功
- 动作必填参数是否齐全
- 执行模式是否允许当前动作（例如 `mock` / `weibo` / `MOBILE_ASSISTED`）

预检输出统一为：

- `allowed`
- `failureCode`
- `message`
- `context`

预检失败时，步骤直接记为失败或阻止创建，具体由调用场景决定，但失败原因必须统一写入日志和状态字段。

### 3. 请求重试规则

将重试逻辑从单条链路中的局部实现，提升为动作级通用策略。

统一策略：

- 只对“可重试错误”进行有限次数重试
- 每个动作定义自己的可重试条件，但走同一套重试执行器
- 固定重试上限，第一版默认最多 2 次额外重试
- 支持动作级退避策略：短等待、长等待
- 任何重试都必须记录在日志中

可重试错误第一版包括：

- 平台显式“系统繁忙”
- 短时网络失败
- 明确可识别的临时风控或服务端异常

不可重试错误第一版包括：

- 参数非法
- 目标不存在
- 账号登录失效
- 权限或内容限制导致的确定性失败

### 4. 成功判定器

为每个动作补齐统一的成功判定器，避免直接用单一布尔值草率判断。

每次动作执行后统一产出：

- `attemptSuccess`：请求是否成功发出并收到可解析返回
- `businessSuccess`：业务动作是否确认成功
- `resultStatus`：`SUCCESS` / `FAILED` / `UNVERIFIED`
- `reason`
- `proof`

说明：

- `SUCCESS`：返回明确成功，或经过校验可确认成功
- `FAILED`：返回明确失败，且不应继续重试
- `UNVERIFIED`：请求结果不够明确，允许调用方结合后续校验策略决定是否二次验证或人工介入

第一版要求：

- `REPOST` 必须继续沿用并统一当前已积累的成功判定经验
- `LIKE`、`COMMENT_LIKE`、`CHECK_IN` 需要补充各自动作的明确判定依据
- `POST` 暂只要求具备统一判定接口，不要求完整真实能力

### 5. 状态回写规则

统一步骤、账号、任务三级状态回写规则，避免不同调用方各写一套。

统一约定：

- 步骤级状态是最小事实来源
- 账号级状态从步骤聚合得出
- 任务级状态从账号级状态聚合得出
- 调用方不直接凭单次请求结果跳过聚合逻辑

推荐状态含义：

- `PENDING`：尚未执行
- `RUNNING`：执行中
- `SUCCESS`：全部成功
- `FAILED`：全部失败或关键步骤失败
- `PARTIAL_FAILED`：部分成功部分失败
- `CANCELLED`：人工取消或流程终止

其中：

- 评论点赞这类“同账号多步骤但失败不中断”的动作，账号最终可能为 `PARTIAL_FAILED`
- 轮转转发这类“同账号失败即停”的动作，账号最终根据已执行步骤汇总得到 `FAILED` 或 `PARTIAL_FAILED`
- 计划执行与互动执行若暂不拆成多步骤，仍需复用统一状态判定接口

### 6. 结构化日志字段

当前日志机制继续沿用 `writeExecutionLog`，但统一字段语义，保证排障可读性。

每次动作执行至少记录：

- `actionType`
- `executorMode`
- `accountId`
- `targetSummary`
- `requestPayload` 摘要
- `responsePayload` 摘要
- `attemptNo`
- `retryCount`
- `success`
- `resultStatus`
- `failureCode`
- `errorMessage`
- `proof` 或判定依据摘要

要求：

- 不记录完整敏感 Cookie
- 对外部返回中的高风险字段做脱敏或摘要化
- 日志中的目标信息必须足够支撑“这是在对哪个对象执行什么动作”的排障需求

## 架构落点

在现有目录结构上补充统一执行稳定性层，不重做执行器。

推荐新增或收敛以下职责：

- `src/server/executors/`：保留具体平台执行器
- `src/server/executors/action-definitions.ts`：集中定义动作配置
- `src/server/executors/target-resolvers.ts`：统一目标解析
- `src/server/executors/result-classifier.ts`：统一成功判定
- `src/server/executors/retry-policy.ts`：统一重试规则
- `src/server/executors/state-writer.ts`：统一状态回写

如果实际实现中发现文件职责可以进一步合并，应优先保持最少文件数量；以上是职责边界，不强制要求一一拆文件。

## 调用链路

统一调用流程如下：

1. 调用方准备动作入参
2. 目标解析器产出结构化目标
3. 动作预检器校验执行条件
4. 统一重试执行器调用具体 executor
5. 成功判定器产出标准结果
6. 状态回写器更新步骤、账号、任务状态
7. 结构化日志写入执行记录

现有 `plan`、`interaction task`、`action job` 三类调用方都应尽量复用该流程，而不是各自实现一套判定逻辑。

## 错误处理

第一版将错误分成以下几类：

- `PRECHECK_FAILED`
- `TARGET_PARSE_FAILED`
- `REQUEST_FAILED`
- `BUSINESS_FAILED`
- `UNVERIFIED_RESULT`
- `ACCOUNT_SESSION_INVALID`
- `RATE_LIMITED`
- `PLATFORM_BUSY`

要求：

- 所有错误都落到统一 failure code 集合中
- 对用户展示时可继续使用友好中文消息
- 对内部日志和测试断言使用稳定错误码

## 兼容策略

- 不移除现有 API 和页面入口
- 不打断现有 `mock executor` 使用方式
- `weibo executor` 优先接入统一稳定性框架
- 已存在的 `runner` 中业务流程保持不变，只逐步替换分散逻辑为统一实现

## 测试与验收

本次至少补齐以下验证：

- 目标解析器：典型 URL 和异常 URL
- 成功判定器：成功、失败、模糊成功、需重试场景
- 重试规则：可重试与不可重试分支
- 状态回写：步骤、账号、任务聚合结果一致
- `mock` 模式下关键链路回归不破坏

静态验收要求：

- `npm run lint`
- `npm run build`

行为验收要求：

- 同一动作在不同调用入口下判定口径一致
- 日志能直接看出重试次数、失败原因和判定依据
- 账号级、任务级状态与步骤事实一致

## 实施顺序

建议按以下顺序推进：

1. 抽离统一错误码和动作配置
2. 抽离目标解析与成功判定器
3. 抽离重试策略
4. 收敛状态回写逻辑
5. 将 `REPOST`、`COMMENT_LIKE`、`LIKE`、`CHECK_IN` 逐步接入
6. 为 `POST` 补统一接入标准和测试桩

## 范围边界

本次不做：

- 真正完整的真实发帖实现
- 全新的任务中心 UI
- 复杂多平台执行抽象
- 高级智能风控策略
- 外部消息告警系统

本次设计目标是先把现有执行链路从“分散修补”提升为“统一稳定框架”。
