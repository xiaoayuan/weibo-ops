# 微博运营台前端独立化与全站视觉改版设计

## 背景

当前项目是一个基于 `Next.js 16 + Prisma + PostgreSQL` 的后台系统，前后端处于同一应用中，页面视觉仍偏基础管理后台风格。用户希望达成两件事：

1. 将整个后台前端改造成深色、稳重、数据运营感更强的管理台风格，并保留浅色主题切换能力。
2. 让前端与后端可以独立启动、独立部署，不再作为一个完全耦合的一体化页面应用继续演进。

本次设计不照搬参考站品牌、文案或业务命名，只借鉴其视觉语言、布局组织和后台信息层级。

## 目标

### 业务目标

- 提升后台整体专业感，弱化“默认脚手架后台”观感。
- 让控制台更适合承载任务执行、账号健康、异常告警、代理状态等运营总览信息。
- 为后续页面扩展建立统一视觉系统，避免各页风格继续分裂。
- 让前端可以脱离 Prisma 和服务端页面逻辑，作为独立 Web 应用演进。

### 工程目标

- 前端与后端可独立 `dev/build/deploy`。
- 后端继续承载现有数据库、调度、执行器、鉴权和业务能力。
- 尽量复用现有 API 和业务逻辑，避免本轮做业务重构。
- 用最小正确改动完成拆分与改版，优先保证主路径可用。

## 非目标

- 不在本轮重写微博执行链路、调度器或 Prisma 数据模型。
- 不在本轮把所有业务组件改造成复杂设计系统库。
- 不在本轮拆成两个完全独立仓库。
- 不在本轮引入新的开放客户端体系或多端 SDK。

## 已确认决策

- 改版范围：全站一次性改版。
- 视觉气质：稳重商务路线，不走强烈赛博炫光风。
- 主题模式：支持深浅双主题，但深色是主视觉标准版。
- 拆分方式：前后端独立运行，推荐同仓双应用架构。
- 参考图用途：只参考风格，不照抄对方品牌名、栏目名或信息内容。

## 总体方案

本次采用“同仓双应用 + 统一设计系统 + 渐进迁移”的方案。

### 目标目录结构

```text
apps/
  web/        # 独立前端应用
  api/        # 独立后端服务
packages/
  shared/     # 共享类型、接口返回结构、常量
docs/
  ...
```

### 方案说明

- `apps/web` 负责页面、路由、主题、交互、前端状态与 UI 组件。
- `apps/api` 负责鉴权、业务 API、Prisma、定时调度、执行器、权限校验。
- `packages/shared` 负责共享类型和接口契约，避免前后端各写一套字段定义。
- 采用同仓而非双仓，减少拆分后共享文档、共享类型和协作成本。

## 视觉与交互设计

### 视觉方向

- 深色背景为主，整体基调偏深蓝黑与石板蓝。
- 重点信息通过青绿色、紫色、红色、黄色等状态色强调，但控制饱和度。
- 卡片、面板、表格、筛选栏统一使用圆角和弱描边，形成清晰层级。
- 通过更强的留白、分区和标题层级提升专业感，而不是依赖过度特效。

### 主题策略

- 默认深色主题，作为主要设计验收标准。
- 同时建立浅色主题语义变量。
- 深浅主题共用同一套语义 token，而不是维护两套互不关联的样式。

### 设计 Token 范围

- 背景色：页面背景、面板背景、悬浮层背景、输入框背景。
- 文本色：主文本、次文本、弱文本、反色文本。
- 描边色：普通边框、强调边框、分割线。
- 状态色：成功、警告、危险、信息、品牌强调色。
- 结构值：圆角、阴影、间距、最大内容宽度。

## 页面系统设计

### 后台壳层

所有后台页面共享统一 Shell：

- 左侧固定导航
- 顶部页面标题栏
- 用户区域
- 主题切换
- 移动端抽屉导航

导航结构保留当前业务信息架构，只重做视觉和选中态、悬浮态、分组关系。

### 页面分类

全站页面统一归为三类：

1. 总览页：控制台、流量监控、调度监控
2. 管理页：账号、超话、任务、计划、日志、代理、用户
3. 配置页：系统设置、风险规则、执行策略、个人资料

### 页面级通用结构

每页尽量收敛为以下骨架：

1. 页面头部：标题、说明、右侧主操作
2. 摘要区：统计卡片或状态摘要
3. 筛选区：搜索、筛选、批量操作
4. 主体区：表格、卡片列表、图表或表单
5. 辅助区：空状态、错误提示、操作反馈

## 组件策略

本轮不引入重型组件库改造，而是围绕现有项目抽出一批可复用外壳组件。

### 需要统一的基础组件

- `PageHeader`
- `StatCard`
- `SurfaceCard`
- `Toolbar`
- `DataTableShell`
- `StatusBadge`
- `FormSection`
- `EmptyState`
- `InlineNotice`
- `ThemeToggle`

### 设计原则

- 优先服务当前页面，避免抽象过度。
- 尽量保留业务组件内部逻辑，只替换结构壳层和视觉表达。
- 表格、表单、筛选区优先统一，收益最大。

## 页面改版范围

### 第一优先级

- 登录页
- 注册页
- 后台整体布局
- 控制台首页

### 第二优先级

- 账号管理
- 超话管理
- 任务配置
- 每日计划
- 控评与轮转
- 执行日志
- 代理中心
- 用户管理
- 系统设置

### 第三优先级

- 文案库
- 互动任务
- 流量监控
- 调度监控
- 其他补齐页面

说明：虽然目标是全站改版，但实施时必须按优先级推进，确保主业务路径先稳定。

## 前后端拆分设计

### 前端职责

- 负责页面展示与交互。
- 不再直接调用 Prisma 或服务端页面级数据读取逻辑。
- 所有业务数据统一通过后端 HTTP API 获取。
- 负责登录页、导航、主题系统、全站样式和页面编排。

### 后端职责

- 保留现有数据库模型和 Prisma。
- 保留现有调度器、执行器、任务生成和日志写入能力。
- 提供稳定 API 给前端消费。
- 负责权限校验、登录态、风控相关校验和服务端业务编排。

### 共享层职责

- 定义接口输入输出类型。
- 定义通用枚举、状态文案映射常量和基础 DTO。
- 避免前后端出现字段不一致。

## 鉴权设计

后台系统更适合使用 Cookie Session 风格，而不是纯前端本地 token 存储。

### 方案

- 登录请求由后端处理并签发登录态。
- 前端通过浏览器自动携带 Cookie 调用 API。
- 后端继续负责鉴权与角色校验。

### 原因

- 适合管理后台。
- 能减少前端自行管理 token 的复杂度。
- 更接近当前系统已有登录方式，迁移风险更低。

## 数据流设计

### 页面数据获取

1. 前端页面加载
2. 前端调用后端 API
3. 后端读取数据库或服务端状态
4. 后端返回结构化 JSON
5. 前端映射到页面组件

### 交互提交流程

1. 用户在前端提交表单或操作
2. 前端调用后端写接口
3. 后端完成验证与业务处理
4. 后端返回结果
5. 前端刷新局部数据并显示反馈

## 错误处理

### 前端

- 页面级错误：显示统一错误面板。
- 表单级错误：显示字段提示或顶部错误提示。
- 操作成功/失败：显示统一 notice/toast 区域或内联提示。
- 对加载中、空状态、失败状态建立统一视觉规则。

### 后端

- API 返回结构统一。
- 对权限不足、参数错误、业务失败、服务器错误进行明确分类。
- 不改变现有关键执行业务逻辑的错误判定语义。

## 响应式设计

- 桌面端使用固定侧栏 + 主内容区。
- 平板和手机端使用抽屉导航。
- 统计卡片自动改单列或双列。
- 表格保持可横向滚动，不牺牲信息完整性。
- 登录和注册页移动端优先保证表单可用性。

## 迁移策略

采用渐进迁移，而非一次性推倒重来。

### 阶段 1：建立新前端壳层

- 建立前端独立应用结构。
- 建立主题变量和通用布局。
- 建立登录、注册、后台导航与控制台首页。

### 阶段 2：迁移核心管理页

- 迁移账号、任务、计划、日志、代理等核心页面。
- 把原页面复用逻辑映射到 API 调用。
- 确保主业务路径可用。

### 阶段 3：补齐剩余页面与统一细节

- 迁移次级页面。
- 统一状态标签、表单节奏、空状态和提示信息。
- 补齐深浅主题下的边角视觉差异。

### 阶段 4：清理旧耦合入口

- 在新前端稳定后，逐步移除旧页面耦合层。
- 保留后端服务端能力，不影响执行链路。

## 风险与缓解

### 风险 1：登录态迁移导致管理端不可用

缓解：优先保持 Cookie 登录态模式，先打通登录、退出、当前用户信息接口。

### 风险 2：页面迁移时接口契约不稳定

缓解：先定义共享类型与 API 返回结构，再迁移页面。

### 风险 3：全站一次性改版导致回归面过大

缓解：按优先级推进，先保证主路径，再补齐其余页面。

### 风险 4：副服务器更新时容易卡死

缓解：部署时采用最小更新策略，优先 `git pull` 与 `docker compose up -d`，仅在必要时才重建镜像，不执行破坏性卷操作。

## 验收标准

### 视觉验收

- 全站视觉统一。
- 深色主题达到主视觉标准。
- 浅色主题可正常切换且不破坏结构。
- 登录页、控制台、列表页、表单页风格一致。

### 功能验收

- 登录、退出、页面导航可用。
- 核心管理页正常读取和提交数据。
- 不破坏现有主要业务交互。
- 后端执行与调度能力不因前端改版受损。

### 工程验收

- 前端与后端可独立启动。
- 前端与后端可独立构建。
- 至少完成静态检查与构建验证。

## 验证方式

- 前端应用执行 lint。
- 前端应用执行 build。
- 后端应用执行必要的构建或校验。
- 对登录、控制台、账号、日志、设置等主路径做人工验证。

## 实施建议

- 优先做“能跑通的新前端壳层”，不要一开始就全量美化所有细节。
- 页面迁移时优先保业务逻辑，视觉细节可第二轮打磨。
- 所有部署遵守当前仓库文档中的安全基线，不做破坏性运维命令。

## 备注

- 当前仓库尚未忽略 `.superpowers/`，如后续持续使用视觉协作文件，应补充到忽略规则中，避免误入版本控制。
- 本设计文档只覆盖前端独立化与全站视觉改版，不包含后续具体任务拆解。
