# `apps/web -> apps/api` 联调核对清单

## 启动前检查

1. 根后端可启动
   - 命令：`npm run dev`
   - 作用：作为仍未完全原生化的兜底后端

2. 独立 API 可启动
   - 命令：`npm run dev:api`

3. 独立前端指向独立 API
   - `apps/web/.env.local`
   - `BACKEND_ORIGIN="http://127.0.0.1:3009"`

4. 独立 API 指向旧后端兜底
   - `apps/api/.env.local`
   - `LEGACY_BACKEND_ORIGIN="http://127.0.0.1:3007"`

5. 数据库与关键环境变量可用
   - `DATABASE_URL`
   - `JWT_SECRET`
   - `ACCOUNT_SECRET_KEY`
   - `EXECUTOR_MODE=weibo`
   - 如需 AI：`AI_API_KEY`、`AI_MODEL`、`AI_BASE_URL`

## 最高优先级

### 1. 登录与鉴权

1. 打开 `/login`
2. 使用管理员账号登录
3. 刷新页面后仍保持登录
4. 打开 `/settings`、`/users`、`/proxy-center` 验证管理员权限
5. 退出登录，再访问后台页应跳回 `/login`

通过标准：
- 登录成功
- Cookie 生效
- 角色权限判断正确

### 2. 账号管理

1. 打开 `/accounts`
2. 新增一个测试账号
3. 编辑账号昵称/分组/状态后保存
4. 用“录入 Session”手动保存 Cookie
5. 点“检测登录态”
6. 如环境允许，再走一次“新增并扫码登录”

通过标准：
- 列表读取正常
- 新增/编辑成功
- 手动录入 Session 成功
- 登录态检测有结果回写
- 扫码链路可生成二维码并轮询状态

风险点：
- `ACCOUNT_SECRET_KEY` 不一致会导致 Cookie 解密失败
- 代理节点异常会影响扫码链路

### 3. 评论池与控评/轮转

1. 打开 `/ops`
2. 手动新增一条评论链接
3. 批量导入评论链接
4. 热评提取
5. 创建控评批次
6. 创建轮转批次
7. 停止一个待执行或执行中的批次

通过标准：
- 评论池 CRUD 正常
- 热评提取返回数据
- 批次创建成功
- 停止成功

风险点：
- 热评提取依赖微博公开接口稳定性
- 真执行依赖账号 Cookie 与代理可用性

### 4. 每日计划

1. 打开 `/plans`
2. 生成指定日期计划
3. 编辑计划时间或文案
4. 通过/驳回某条计划
5. 停止某条计划
6. 执行一条计划

通过标准：
- 计划生成成功
- 编辑、通过、驳回、停止成功
- 执行请求能返回结果并写日志

风险点：
- `FIRST_COMMENT` 现在是最小真实闭环，优先重点验证
- 其余计划类型虽然已接真实执行器，但仍建议逐类型点验

### 5. 互动任务

1. 打开 `/interactions`
2. 创建点赞任务
3. 创建回复任务
4. 批量删除
5. 通过/驳回/停止
6. 执行一条互动任务

通过标准：
- 列表读取正常
- 批量创建成功
- 停止与审批成功
- 执行可返回结果并写日志

## 第二优先级

### 6. 文案库与 AI

1. 打开 `/copywriting`
2. 保存 AI 接口配置
3. 保存 AI 风险词配置
4. 生成候选文案
5. 改写候选文案
6. 预审候选文案
7. 保存选中文案

通过标准：

- 生成/改写/预审/保存可用
- 链接预览返回内容

风险点：
- AI 接口地址受校验限制
- 无 `AI_API_KEY` 时相关功能会失败

### 7. 日志、流量、调度快照

1. 打开 `/logs`
2. 验证筛选、汇总/明细切换
3. 触发 AI 日志总结
4. 打开 `/traffic`
5. 打开 `/scheduler`

通过标准：
- 日志读取得到数据
- AI 总结返回结果
- 流量页和调度页可读

### 8. 设置页

1. 打开 `/settings`
2. 修改密码
3. 修改代理配置
4. 测试代理
5. 保存执行策略
6. 保存风控规则
7. 查看执行器状态

通过标准：
- profile 保存成功
- 代理测试成功或返回明确失败原因
- 策略和规则保存成功
- 执行器状态可读

通过标准：

## 高风险点清单

1. 账号 Cookie 与解密密钥不一致
   - 现象：登录态检测失败、扫码后保存失败

2. 执行器环境变量不一致
   - 重点看 `EXECUTOR_MODE`

3. 代理相关环境差异
   - 扫码登录、真实执行、热评提取都可能受影响

4. AI 接口可用性
   - 文案生成、AI 风控、日志总结都依赖 AI 服务

5. 计划执行类型差异
   - `FIRST_COMMENT` 优先重点验证
   - `LIKE/REPOST/COMMENT/CHECK_IN` 逐个点验

## 建议验证顺序

1. 登录
2. 账号管理
3. 评论池与控评/轮转
4. 每日计划
5. 互动任务
6. 文案库与 AI
7. 设置页
8. 日志/流量/调度
9. 其他管理页

## 当前结论

当前系统已经适合进入一轮完整联调。

联调时最值得优先关注的不是普通 CRUD，而是：

1. 扫码登录是否稳定
2. 手动 Cookie 录入后登录态检测是否稳定
3. `FIRST_COMMENT` 是否能真实完成
4. 互动回复是否能真实完成
5. 控评与轮转创建后停止链路是否一致
